v0.5.0 · Design Note

Hooking model internals under a FULL CUDA graph

Capturing attention, hidden states, and steering the residual stream — while the step is a frozen sequence of GPU kernels and no Python ever runs again. We turn cudagraph_mode=FULL on and let vLLM decide, per step, whether to replay a graph or run eagerly; our hook is correct either way, and faster than the eager hook it replaces.

00The whole idea, in one picture

Every hook is split into two halves. Placement is static — a custom kernel baked onto candidate layers at capture time. Behavior is data — the bytes of pre-registered GPU buffers that we overwrite before each step and the baked kernel reads while it runs. That single split is what lets a hook live inside a graph that can never run Python again: the graph is frozen, but the bytes it reads are not.

01FULL mode: we don't choose graph-vs-eager — vLLM does

We run with cudagraph_mode=FULL. At compile time vLLM captures a full CUDA graph for a fixed set of batch sizes (cudagraph_capture_sizes). At runtime, every step — prefill, mixed, or decode — vLLM's dispatcher pads the batch to the nearest captured size and replays that graph; if the batch is larger than any captured size (a big prefill chunk, an odd shape) it falls back to eager. We never make that choice and we no longer hard-split “eager prefill + graph decode” — vLLM decides per step, by size, and our hook is correct on both paths.

02Why the eager hook cannot survive a graph

The v0.2.0 hook is a register_forward_hook callback — a Python function that fires after a module's forward. torch.compile traces each forward once and replaces it with a compiled artifact; the instance hook is never called again. And a captured CUDA graph is a frozen replay of GPU kernels — no Python runs on replay at all. So the eager hook silently captures nothing under a graph. By default the plugin therefore forces enforce_eager=True; arming VLLM_HOOK_ALLOW_CUDAGRAPH=1 selects the graph-safe path below.

The constraint everything follows from On replay, the step is data-in → data-out through fixed kernels. Any per-request logic must be expressed as bytes the kernels read, written before the replay — never as code that runs during it.

03Placement: a baked op, not a kernel patch

We do not modify FlashAttention or any vLLM CUDA kernel. We register our own custom ops — vllm_hook::capture_qk, capture_hs (read-only scatter) and steer_buffer (in-place residual add) — via torch.library, and attach them by class-wrapping the Python .forward of the Attention module (for QK) or the decoder layer (for HS): a reversible monkey-patch that calls our op right before / after the original forward. vLLM's attention runs byte-for-byte unchanged; capture only reads the same post-RoPE q/k it is handed, so the model's numerics are identical with or without the hook.

  • Hosts (built at load_model, before capture, so data-ptrs are fixed): one per hooked layer, owning the static buffers.
  • The op is never a splitting op — that is what lets it be absorbed into the full graph and replay (§01). The capture/steer-mode env var gates the declaration per worker, so each worker keeps a distinct compile cache key.

04Behavior: the kernel reads a table we rewrite each step

The baked op is a fixed instruction — “read this index buffer, and for each token do X to the row it names.” What it actually does on a given step is decided entirely by the integers we wrote into that buffer beforehand.

Capture: a routing index + a discard sentinel

There is a capture_index buffer with one slot per token position. Before each step the host fills it — “token in column 3 → buffer row 5” — and crucially row 0 is a write-only discard sentinel: any token we do not want is pointed at row 0 and harmlessly overwritten. So “capture layer 7 for request B but not request A” is just integers: B’s tokens get real rows, A’s tokens get 0 — no branch, same kernel, no cost difference on replay.

Steering: a coefficient and a vector id

The buffers are coeff[layer, token] and vec_id[layer, token] plus a resident vec_table[V_max, hidden]; the baked kernel does residual += coeff · vec_table[vec_id], masked and in place. Setting coeff = 0 makes it a numeric no-op (steering off); setting coeff = c, vec_id = v across request r’s tokens steers that request by vector v at strength c. A per-token steer_mode mask unifies both methods — add_vector (host coefficient) and adjust_rs (coefficient computed in-kernel from the live residual) — so one batch can mix them (full diagram in §09).

05Where the data and metadata live

Three planes. Routing/config metadata for the active hook layers lives in device slabs, staged from a pinned host ring. Captured data lands first in static GPU scratch, then is cloned into per-request host-side buckets (tensors still on GPU, the .cpu() deferred), and finally materializes as CPU artifacts at request end. The only source of per-request intent is each request’s extra_args in its sampling params, read on the host during routing.

Sizing the buffers — the equation

Every static buffer is allocated once at load_model (before capture, so its data-ptr is fixed) and sized from model dims and a single scheduler knob. The driver is cap = max_num_batched_tokens — the largest single-step token count — not max_model_len: flat per-step routing only ever stages one step’s tokens, so the buffer is sized to one batch, not the whole sequence.

# per worker, summed over the L hooked layers · b = dtype bytes (2 fp16/bf16, 4 fp32)
QK    bytes = L · (cap + 1) · (q_dim + k_dim) · b
HS    bytes = L · (cap + 1) · d_model · b
route bytes = L · cap · 8           # int64 capture_index (+ tiny any_active)
steer bytes = V_max · d_model · b   # vec_table (+ per-(layer,token) coeff/vec_id slabs)

  cap     = max_num_batched_tokens   # the +1 row is the row-0 discard sentinel
  q_dim   = n_heads    · d_head
  k_dim   = n_kv_heads · d_head      # grouped-query: usually < q_dim
  d_model = hidden size  ·  L = hooked layers  ·  V_max = steering vector-table capacity

Example — Qwen2-1.5B (L=28, q_dim=1536, k_dim=256, fp32) at cap=2048: 28 · 2049 · (1536+256) · 4 ≈ 392 MiB of QK scratch — exactly what the worker logs. It scales linearly in cap, so cap=16384 is 8× that. Densifying cudagraph_capture_sizes changes how many steps replay, not this footprint.

06The per-step workflow: prefill & decode

Every step has the same four beats, bracketing the forward from outside the graph: a host routing build, the (replayed or eager) forward where the baked op fires, and a host egress drain. Routing is wired into a wrap of the model runner’s _prepare_inputs — vLLM calls it inside synchronize_input_prep(), so we read this step’s query_start_loc with no added sync. Prefill and decode differ only in how much each beat does.

The whole machine, in one sentence The slab is a per-step address label the CPU writes ("the token in column 2 goes to row 3"); the buffer is the fixed mailbox grid the GPU kernel drops rows into by following those labels; row 0 is the trash slot; the view is how each layer sees its own row of labels. Construction lays out the mailboxes once; every step we only rewrite the labels and sweep the delivered mail into per-request bins. Nothing is ever sized to the sequence — only to one step, which is cap by definition.

08Capture data path: flat routing & the growing K history

The hidden-state artifact is simple — the per-token residual stream is the artifact, so egress just clones each step’s rows. QK has a wrinkle: k_all for a token is the full key history up to it. We do not accumulate that history in the buffer; we capture it incrementally and rebuild it at flush.

  • Flat per-step routing. Each token routes to row flat_column + 1 — its position within this step. A step’s token count is bounded by cap = max_num_batched_tokens, so a step can never write past the buffer.
  • Incremental egress (W3). Egress clones only this step’s new key rows (O(1) per decode step instead of re-cloning the whole O(N) prefix) and records the absolute prefix length in k_prefix_ends.
  • Flush-time reconstruction. At request end we concatenate the per-step clones and slice the growing prefixes — [full[:L] for L in k_prefix_ends] — reproducing exactly what the eager path would have stored, byte for byte.
Why per-step, not absolute position The earlier scheme routed by absolute sequence position and sized the buffer to one batch. A prompt longer than cap (i.e. any chunked prefill) then dropped every token past cap — a silent under-capture. Flat per-step routing fits every step by construction, so a long / chunked prompt is captured in full; and because each request now owns a disjoint column range, several requests can share the buffer without colliding. Proven against eager with a forced-chunked-prefill oracle, including a no-chunk control that isolated the small residual diff as the engine’s own compiled-vs-eager fp matmul drift, not a capture error.

09Steering: the mutating counterpart both methods

Capture is read-only; steering must change the residual so the edit reaches every downstream layer. The op declares mutates_args=["residual"] and edits in place — a functional variant that returned a new tensor was proven on GPU to fire but have its return value dropped across a segment seam; in-place + the mutates contract is what threads the edit downstream.

Four optimizations, all riding the buffer-mode discipline:

#OptimizationWhat it removes
W1Invalidation routingWhen the batch signature and upload width are unchanged, the device slabs already hold an identical routing — skip reset/build/upload. Steady-state steering decode becomes a pure replay (its key is bit-identical each step).
W2Pinned-mirror ring (depth 2)The per-step CPU↔GPU sync a single mirror would force: reset waits on a slot whose H2D upload finished steps ago, not the in-flight one. Routing→forward stops being serialized.
W3Incremental egressThe O(N²) per-step re-clone of the whole k_all prefix — now O(1) rows/step + flush-time reconstruction (§08). O(N) memory, byte-identical artifact.
W4Budget-driven drainsGPU residency under heavy capture: drain GPU→pinned host on a dedicated copy stream when over a byte budget (default off → zero behavior change).
