v0.5.4 · Design Note

Hooking model internals under a FULL CUDA graph

Capturing attention, hidden states, and steering the residual stream — while the step is a frozen sequence of GPU kernels and no Python ever runs again. We turn cudagraph_mode=FULL on and let vLLM decide, per step, whether to replay a graph or run eagerly; our hook is correct either way, faster than the eager hook it replaces, and — under high-throughput continuous batching — bounded in memory so capture does not OOM the engine.

00The whole idea, in one picture

Every hook is split into two halves. Placement is static — a custom kernel baked onto candidate layers at capture time. Behavior is data — the bytes of pre-registered GPU buffers that we overwrite before each step and the baked kernel reads while it runs. That single split is what lets a hook live inside a graph that can never run Python again: the graph is frozen, but the bytes it reads are not.

01FULL mode: we don't choose graph-vs-eager — vLLM does

We run with cudagraph_mode=FULL. At compile time vLLM captures a full CUDA graph for a fixed set of batch sizes (cudagraph_capture_sizes). At runtime, every step — prefill, mixed, or decode — vLLM's dispatcher pads the batch to the nearest captured size and replays that graph; if the batch is larger than any captured size (a big prefill chunk, an odd shape) it falls back to eager. We never make that choice and we no longer hard-split “eager prefill + graph decode” — vLLM decides per step, by size, and our hook is correct on both paths.

02Why the eager hook cannot survive a graph

The v0.2.0 hook is a register_forward_hook callback — a Python function that fires after a module's forward. torch.compile traces each forward once and replaces it with a compiled artifact; the instance hook is never called again. And a captured CUDA graph is a frozen replay of GPU kernels — no Python runs on replay at all. So the eager hook silently captures nothing under a graph. By default the plugin therefore forces enforce_eager=True; arming VLLM_HOOK_ALLOW_CUDAGRAPH=1 selects the graph-safe path below.

The constraint everything follows from On replay, the step is data-in → data-out through fixed kernels. Any per-request logic must be expressed as bytes the kernels read, written before the replay — never as code that runs during it.

03Placement: a baked op, not a kernel patch

We do not modify FlashAttention or any vLLM CUDA kernel. We register our own custom ops — vllm_hook::capture_qk, capture_hs (read-only scatter) and steer_buffer (in-place residual add) — via torch.library, and attach them by class-wrapping the Python .forward of the Attention module (for QK) or the decoder layer (for HS): a reversible monkey-patch that calls our op right before / after the original forward. vLLM's attention runs byte-for-byte unchanged; capture only reads the same post-RoPE q/k it is handed, so the model's numerics are identical with or without the hook.

  • Hosts (built at load_model, before capture, so data-ptrs are fixed): one per hooked layer, owning the static buffers.
  • The op is never a splitting op — that is what lets it be absorbed into the full graph and replay (§01). The capture/steer-mode env var gates the declaration per worker, so each worker keeps a distinct compile cache key.

04Behavior: the kernel reads a table we rewrite each step

The baked op is a fixed instruction — “read this index buffer, and for each token do X to the row it names.” What it actually does on a given step is decided entirely by the integers we wrote into that buffer beforehand.

Capture: a routing index + a discard sentinel

There is a capture_index buffer with one slot per token position. Before each step the host fills it — “token in column 3 → buffer row 5” — and crucially row 0 is a write-only discard sentinel: any token we do not want is pointed at row 0 and harmlessly overwritten. So “capture layer 7 for request B but not request A” is just integers: B’s tokens get real rows, A’s tokens get 0 — no branch, same kernel, no cost difference on replay.

Steering: a coefficient and a vector id

The buffers are coeff[layer, token] and vec_id[layer, token] plus a resident vec_table[V_max, hidden]; the baked kernel does residual += coeff · vec_table[vec_id], masked and in place. Setting coeff = 0 makes it a numeric no-op (steering off); setting coeff = c, vec_id = v across request r’s tokens steers that request by vector v at strength c. A per-token steer_mode mask unifies both methods — add_vector (host coefficient) and adjust_rs (coefficient computed in-kernel from the live residual) — so one batch can mix them (full diagram in §10).

05Where the data and metadata live

Three planes. Routing/config metadata for the active hook layers lives in device slabs, staged from a pinned host ring. Captured data lands first in static GPU scratch, then is cloned into per-request host-side buckets (tensors still on GPU, the .cpu() deferred), and finally materializes as CPU artifacts at request end. The only source of per-request intent is each request’s extra_args in its sampling params, read on the host during routing.

06Buffers & how to size them Granite-3.1-8B worked example

Everything the graph reads or writes is allocated once at load_model — before vLLM compiles or captures, so every data_ptr is frozen for the lifetime of the replays. There are four families of fixed allocation, sized purely from model dims and a single scheduler knob. (The fifth memory plane, the per-request artifact buckets, is not pre-sized — it grows with decode and is bounded at runtime; see §11.)

The one knob that drives every size

The driver is cap = max_num_batched_tokens — the largest number of tokens one step may carry — not max_model_len. Flat per-step routing (§09) only ever stages one step’s tokens into the buffers, so a buffer is sized to a single batch, never to a whole sequence. The static buffers also carry a +1 row: row 0 is the write-only discard sentinel.

The sizing functions

# model dims (b = dtype bytes: bf16/fp16 → 2, fp32 → 4)
b       = dtype_bytes          # element size of the model's activations
q_dim   = n_heads    * d_head  # query width
k_dim   = n_kv_heads * d_head  # key width (GQA: usually < q_dim)
hidden  = d_model              # residual width
cap     = max_num_batched_tokens   # the +1 row is the row-0 discard sentinel

def qk_scratch_bytes(L, cap, q_dim, k_dim, b):     # ① QK worker
    return L * (cap + 1) * (q_dim + k_dim) * b

def hs_scratch_bytes(L, cap, hidden, b):          # ① HS worker
    return L * (cap + 1) * hidden * b

def route_slab_bytes(L, cap):                     # ② QK + HS
    return L * cap * 8  +  L * 4        # capture_index int64 + any_active int32

def steer_slab_bytes(L, cap):                     # ③ steer slabs
    return L * cap * (4 + 8 + 8)         # coeff fp32 + vec_id int64 + mode int64

def steer_table_bytes(V_max, hidden, b):          # ③ resident vector library
    return V_max * hidden * b  +  V_max * 4

def pinned_ring_bytes(per_step_slab_bytes, ring_depth=2):   # ④ host staging
    return ring_depth * per_step_slab_bytes

07The per-step workflow: prefill & decode

Every step has the same four beats, bracketing the forward from outside the graph: a host routing build, the (replayed or eager) forward where the baked op fires, and a host egress drain. Routing is wired into a wrap of the model runner’s _prepare_inputs — vLLM calls it inside synchronize_input_prep(), so we read this step’s query_start_loc with no added sync. Prefill and decode differ only in how much each beat does.

09Capture data path: flat routing, growing K & the cached prefix

The hidden-state artifact is simple — the per-token residual stream is the artifact, so egress just clones each step’s rows. QK has a wrinkle: k_all for a token is the full key history up to it. We do not accumulate that history in the buffer; we capture it incrementally and rebuild it at flush.

  • Flat per-step routing. Each token routes to row flat_column + 1 — its position within this step. A step’s token count is bounded by cap = max_num_batched_tokens, so a step can never write past the buffer.
  • Incremental egress (W3). Egress clones only this step’s new key rows (O(1) per decode step instead of re-cloning the whole O(N) prefix) and records the absolute prefix length in k_prefix_ends.
  • Flush-time reconstruction. At request end we concatenate the per-step clones and slice the growing prefixes — [full[:L] for L in k_prefix_ends] — reproducing exactly what the eager path would have stored, byte for byte.
Why per-step, not absolute position The earlier scheme routed by absolute sequence position and sized the buffer to one batch. A prompt longer than cap (i.e. any chunked prefill) then dropped every token past cap — a silent under-capture. Flat per-step routing fits every step by construction, so a long / chunked prompt is captured in full; and because each request now owns a disjoint column range, several requests can share the buffer without colliding. Proven against eager with a forced-chunked-prefill oracle, including a no-chunk control that isolated the small residual diff as the engine’s own compiled-vs-eager fp matmul drift, not a capture error.

The prefix-cache hole — keys that were never forwarded v0.5.2

Flat routing fixed the chunked-prefill drop, but a continuous-batching server with enable_prefix_caching=True exposes a second, subtler under-capture. When two requests share a prompt prefix, the scheduler does not recompute the shared part: the request starts at num_computed_tokens = C (the cached length) and only the tail is forwarded. Those C cached keys never pass through attention, so the static buffer never sees them — buffer mode would capture only the ~tail, where eager reconstructs the whole history from the paged KV cache mid-forward.

The fix lives entirely at egress (post-forward, off-graph — no in-forward stash, no graph break). On the first capture step for a (request, layer) where num_computed > 0, we read the [0, C) cached keys straight from the persistent paged KV cache — static_forward_context[name].kv_cache indexed by the request’s block_table row — and prepend them as k_all[0]. The flush-time full[:abs_end] reconstruction then yields the complete key history, byte-identical to eager. When there is no prefix cache (num_computed == 0) it is a strict no-op, so the FULL-milestone tests cannot regress. Validated on LSF by running the same prompt twice (run 2 is a guaranteed cache hit): num_cached=240 + 16 forwarded = 256 = the full prompt, matching eager.

HS is deliberately not changed: residuals live in no cache, so neither eager nor buffer can reconstruct a prefix-trimmed residual — they are equally limited and parity holds.

10Steering: the mutating counterpart both methods

Capture is read-only; steering must change the residual so the edit reaches every downstream layer. The op declares mutates_args=["residual"] and edits in place — a functional variant that returned a new tensor was proven on GPU to fire but have its return value dropped across a segment seam; in-place + the mutates contract is what threads the edit downstream.

11Memory under load: continuous drains & admission throttle v0.5.3–0.5.4 · opt-in

The fixed buffers of §06 are bounded by construction. The one plane that is not is the per-request artifact bucket (§05): egress clones each step’s rows GPU→GPU and defers the .cpu() to request completion, so live residency is Σin-flight reqs(layers × tokens × dim) and grows with both concurrency and decode length. Under high-reqs/s continuous batching that overruns memory and the engine OOMs. The same retention policy is shared verbatim by the v0.2.0 eager path — graph mode only shifts where the OOM lands.

The cost, made concrete For Granite-8B QK in all_tokens mode, one in-flight request at a 2048-token context holds roughly the same bytes as the entire static QK scratch — ≈640 MiB (L · (P+T) · (q_dim+k_dim) · b). At concurrency N that is N× that, held until each request flushes. The clones themselves are cheap; holding them all is the problem.

Two opt-in stages, armed by VLLM_HOOK_CAPTURE_STREAMING=1 (default OFF → the manager is inert and the parity oracles are byte-for-byte unchanged):

v0.5.4 — the trigger had to track current residency, not throughput The drain trigger gpu_bytes was first a monotonic cumulative counter: it accrued at every egress but was never decremented on flush, so it crossed the budget from sheer throughput even on a fixed-shape workload with no memory pressure (num_preemptions = 0, KV ≤ 10%). Those drains relieved nothing yet cost a periodic burst of pinned-alloc + D2H on the launch thread — a measured +38–44% TPOT regression for QK last_token at N16, scaling with capture volume. The fix makes gpu_bytes track current GPU residency: charged the per-request reduced bytes at egress, reset to 0 on drain, and decremented on on_pop by the bucket’s still-GPU bytes (a host-drained tensor already left the GPU, so it must not re-credit). On fixed-shape, live residency stays well under budget → drains never fire → the regression is gone; under real continuous-batching pressure residency climbs to the budget → drains fire → OOM relief is preserved.
KnobDefaultEffect
VLLM_HOOK_CAPTURE_STREAMING0 (off)Master opt-in — enables both the drain budget and the resident-ceiling throttle.
VLLM_HOOK_CAPTURE_GPU_BUDGET0.05 × free GPUFraction of free GPU the clones may occupy before a drain fires (Stage 1). Explicit 0 disables the drain.
VLLM_HOOK_CAPTURE_DRAIN_RING3Depth of the drain event ring, so continuous draining waits a stale event, not the in-flight D2H.
VLLM_HOOK_CAPTURE_RESIDENT_CEILING(_FRAC)0.5 × host RAMTotal resident-byte ceiling (Stage 3). Over it, new requests are refused; ≤0 = no ceiling.

A pooled/shared pinned arena was considered and rejected: sharing a drain buffer across requests breaks the ceiling’s byte accounting (a shared buffer lives until its last view frees, so resident_bytes would under-count real host RAM). The per-tensor fresh torch.empty(pin_memory=True) is load-bearing for accurate accounting. Stage 2 (incremental sink + free to the async-save thread, collapsing the host retention window to N steps) is deferred — Stage 1 + Stage 3 already prevent OOM.

The decode-perf wins ride four optimizations on the buffer-mode discipline:

#OptimizationWhat it removes
W1Invalidation routingWhen the batch signature and upload width are unchanged, the device slabs already hold an identical routing — skip reset/build/upload. Steady-state steering decode becomes a pure replay (its key is bit-identical each step).
W2Pinned-mirror ring (depth 2)The per-step CPU↔GPU sync a single mirror would force: reset waits on a slot whose H2D upload finished steps ago, not the in-flight one. Routing→forward stops being serialized.
