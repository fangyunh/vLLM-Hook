v0.4.0 · Design Note

Hooking model internals under a FULL CUDA graph

Capturing attention, hidden states, and steering the residual stream — when the decode step is a frozen sequence of GPU kernels and no Python ever runs again.

00The whole idea, in one picture

Every hook gets split into two halves. Placement is static — a kernel baked onto candidate layers at capture time. Behavior is data — the contents of pre-registered GPU buffers that we overwrite before each replay and a baked kernel reads during replay. That single split is what lets a hook live inside a graph that can never run Python again.

01A 30-second primer on CUDA-graph modes

A CUDA graph records a sequence of GPU kernels once and replays them with one cheap launch — eliminating per-kernel Python/launch overhead. vLLM exposes several modes that differ in how much of the forward pass is frozen into graphs, and therefore where Python is still allowed to run.

02Why the v0.3.0 hooks cannot survive FULL

The shipping (PIECEWISE) hook is op mode: a custom op (vllm_hook::qk_probe, hs_probe, steer_residual) registered as a vLLM splitting op, which forces an eager seam around it. Its Python impl runs mid-forward, exactly where the original eager hook lived — and it does real interpreted work:

def _qk_probe_impl(q, k, sink, layer_idx):
    _bump_sink(sink)
    capturing = torch.cuda.is_current_stream_capturing()
    if capturing:
        return                  # ← bails during capture / replay
    fn(q, k, int(layer_idx))     # ← get_forward_context(), req_ids walk, .item(), dict lookups

Under PIECEWISE this is fine: the splitting op makes the layer an eager seam that runs every step, so the impl fires on decode too (that seam is also the source of the ~6.5% wall-time tax). Under FULL there is no seam. The op is absorbed into the captured graph, and on replay the kernels run with no Python at all — the entire decision layer is in the one place FULL never visits.

The real blocker It was never "FULL has no interrupts so we must hook every layer." It is deeper: replay re-runs frozen kernels over whatever bytes are in their buffers, and Python is gone. Any design that needs to decide something at decode time is disqualified. The decision has to become data.

03The reframe: placement is static, behavior is data

Once you accept that replay reads buffers and runs no Python, the design writes itself. Split every hook into the two halves from §00:

HalfWhen decidedHow expressedExample
Placementcapture time (static)which layers carry a baked opbake capture/steer on every candidate layer
Behaviorper step (dynamic)contents of pre-registered GPU buffers, read by the baked kernelcoeff=0 → no-op; route to sentinel row → discard

This dissolves the "hooking every layer is wasteful" worry. In PIECEWISE the expensive thing was the seam, not the op. In FULL there are no seams, and a data-gated op on a layer nobody is using is a masked add with coefficient 0 — a few microseconds, fully hidden behind the kernels around it.

Counter-intuitive but true FULL does not make hooks more expensive — it makes them static and cheap. Over-provision the candidate set; you pay almost nothing for the unused entries.

The precedent we copy: LoRA under cudagraphs

vLLM already does per-request, cudagraph-safe, mixed-batch intervention. It pre-registers a fixed number of adapter slots at a fixed max rank, bakes the LoRA GEMM into the graph for all slots, keeps a per-row token → adapter-slot mapping buffer, and gates no-adapter rows to zero. A steering vector is literally a rank-1 additive LoRA on the residual stream. The batching question is already answered in shipping code; we mirror its shape.

04The fix: buffer mode

Buffer mode is the concrete realization of "behavior is data." Three pieces, with a hard line drawn between what is inside the graph and what is outside it.

Why one mechanism covers both phases Buffer mode does not care whether the forward is eager or graphed. The scatter kernel runs eagerly during prefill and replays during decode; routing and egress are host Python in the wrapper either way. So we do not mix op-mode-for-prefill with buffer-mode-for-decode — the same path serves both, and it declares no splitting op, so the decode graph stays one full graph.

Two memory planes — and why SHM does not help here

It is easy to conflate two different "buffers." Keep them apart:

PlaneWhat it isMemory spaceExamples
In-graph data planebuffers the captured kernels read/write on replayCUDA device memory, fixed data_ptrq_buf/k_buf, capture_index, coeff, vec_table
Egress / transport planehow drained results travel worker → driverhost RAM / disk / host SHMrpc, disk, shm

The fixed pre-defined buffer the FULL design depends on is the in-graph data plane — it must be device memory, because a recorded kernel scatters into it by data_ptr. A graph kernel cannot touch host shared memory, so VLLM_HOOK_USE_SHM simply cannot be that buffer. On the egress side where SHM does live, it is already dominated (disk-st-async beats it on every measured cell, and it is HS-only / last-token-only). The legitimate concern hiding in the SHM question — egress throughput when decode capture drains every step — is answered not by a new transport but by accumulating rows in the device buffer and draining in batches, with pinned host memory for the D2H copy.

05Capture, end-to-end implemented · parity-proven

Capture is the milestone that is built and validated. Two ops, the same shape: capture_qk (attention Q/K) and capture_hs (decoder-layer residual stream). Each scatters live rows into a static per-layer buffer keyed by a routing index.

The scatter, concretely

The static buffer is sized (cap + 1, dim). Row 0 is a write-only discard sentinel: any token we do not want to capture is routed there and harmlessly overwritten. That sentinel is what makes "baked but inactive" a true no-op with zero branching.

06The target: eager prefill + full-graph decode

Under cudagraph_mode=FULL_DECODE_ONLY, a request's life looks like this. Notice the scatter kernel is the same kernel in both phases; only its execution mode differs (executed eagerly vs. replayed). Routing and egress bracket every step from outside the graph.

07Steering: the mutating counterpart implemented · parity-proven

Capture is read-only; steering must change the residual stream so the edit propagates through every downstream layer. The PIECEWISE version (steer_residual) runs its math in a Python impl at the seam — which, per §02, FULL skips. The FULL version (VLLM_HOOK_STEER_MODE=buffer) applies the very same buffer-mode discipline to a mutating kernel: this is the LoRA-slot model made literal. It is built and parity-proven for the add_vector method.

Why this is the milestone that answers the user's worry Toggling coeff per step and per slot and getting token-for-token parity vs eager is the proof that a frozen FULL graph can honor per-request configuration — by data alone, with no Python on replay.

08Per-request config & the frozen-menu contract

Continuous batching mixes requests with different hook configs in one decode graph. The buffer approach handles this directly: keep a persistent CPU + GPU mirror of the hook config, keyed by the stable batch slot (input_batch.req_ids[i] ↔ row i — the same indexing the routing already uses).

The contract this imposes — state it to users up front

Frozen menu The candidate hookable layers, the max concurrent request slots, the vector-table capacity (V_max), and the worst-case capture scope are frozen at capture time. Per request you may pick anything within the menu (which menu-layer, which menu-vector, which coefficient, capture-or-not). Anything outside the menu needs a re-capture (seconds, offline-ish).

For genuinely unbounded cases, route them off the FULL path: in-spec requests run on the FULL graph; out-of-spec requests fall back to PIECEWISE or eager (vLLM already keeps an eager path for uncaptured batch sizes — we piggyback the dispatch). A clean profiling metric drops out for free: the graph-eligible request fraction.

09What is proven, and what is next

Every cell is validated by the established subprocess-isolated graph-vs-eager numerical oracle on LSF (capture → eager → compare), Qwen2-1.5B, rtol=atol=1e-2. The two engines must run in separate processes because the GPU is exclusive_process.
